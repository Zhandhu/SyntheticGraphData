# Tongyi Data Generation Pipeline

Reconstruction and extension of the synthetic QA dataset generation pipelines from two Alibaba/Tongyi Lab papers, implemented as a modular async Python codebase.

**Source papers:**
- Track A: [WebFrontier](https://arxiv.org/abs/2509.13309) (Qiao et al., 2025) — corpus-anchored seed QA pairs iteratively escalated by a ReAct agent
- Track B: [Tongyi DeepResearch](https://arxiv.org/abs/2510.24701) (2025) — entity-anchored knowledge graph with subgraph-grounded QA synthesis

Track B is the primary implementation. All commands below operate on Track B unless noted. See `report.tex` for full design rationale and implementation details.

---

## How It Works

Questions are grounded in a knowledge graph (KG) of atomic retrieved facts. Every KG node is a single fact from one tool call, traceable to its source. Questions are generated by sampling a subgraph and asking the LLM to connect its nodes — so correctness is verifiable against the graph, which is verifiable against the retrieved sources.

**Pipeline per batch:**
```
KG (built once)
  → SubgraphSampler     (random walk or bridging → connected subgraph)
  → QA generation       (LLM writes question spanning ≥3 entities)
  → UncertaintyInjector (atomic reframings within existing subgraph)
  → GraphEscalator      (ReAct loop extends subgraph by 1-hop, rewrites question)
  → QualityControl      (base solver → advanced solver → judge → dedup)
  → passing_qas.jsonl + passing_trajectories.jsonl
```

**Three difficulty levers** (in order of impact):
1. **Sampling** (primary) — `walk_length` and `bridge_components` set the topological ceiling. Escalation cannot exceed this.
2. **Uncertainty injection** (secondary) — reframes within the existing subgraph; harder to parse, same knowledge required.
3. **Escalation** (additive) — adds at most 1 hop per round from whatever sampling produced.

## Quick Start

### 1. Build the Knowledge Graph

The KG is expensive to build (~30 min, GPT-4o) but built once and reused indefinitely.

```bash
# Full rebuild from all seed topics in data/seeds/seed_entities.json
python scripts/rebuild_kg.py
# → data/output/knowledge_graph.json  (205 entities, 1106 relations, 1 component)

# Add new seed topics to an existing KG without full rebuild
python scripts/expand_kg.py --seeds "new topic 1" "new topic 2"
```

Current seeds span 5 domains — science, history, economics, technology, geopolitics — with ~57 seed entities. See `data/seeds/seed_entities.json`.

### 2. Generate QA Pairs

```bash
# Original pipeline (fixed parameters, no complexity scaling)
python scripts/generate_until_pass.py --batches 5 --batch-size 8

# With complexity control (0.0 = simple, 1.0 = maximally complex)
python scripts/generate_until_pass.py --complexity 0.7 --batches 5 --batch-size 8

# Output (appended across runs, not overwritten):
#   data/output/passing_qas.jsonl
#   data/output/passing_trajectories.jsonl
#   data/output/ui_data.json  (viewer bundle)
```

### 3. Visualiser (Optional)

```bash
python ui/serve.py
# → http://localhost:8765/ui/viewer.html
```

The viewer shows the KG as a D3.js force-directed graph, highlights the subgraph for each QA pair, and provides a generation panel to run new batches directly from the browser.

## Complexity Control

`--complexity c` (float, 0.0–1.0) linearly scales generative parameters. Omitting it runs the original fixed pipeline unchanged.

| Parameter | c=0 | c=1 | Effect |
|---|---|---|---|
| `walk_length` | 10 | 30 | Primary diameter lever |
| `walk_min` / `walk_max` | 2–5 | 6–12 | Subgraph size range |
| `bridge_components` | 2 | 4 | Cross-domain span |
| `bridge_per_comp` | 2 | 4 | Neighborhood density per cluster |
| `max_tool_calls_qc` | 8 | 12 | Advanced solver budget |
| escalation rounds | 1 | 1 | Fixed (cost/gain tradeoff) |

Strategy weights shift toward `subgraph_bridging` and `multi_entity_aggregation` at high cx. Formula: `w_i(c) = max(0.05, a_i * c + (1-a_i) * (1-c))` where affinity `a_i` per strategy: hop\_extension=0.1, subgraph\_bridging=0.8, multi\_entity\_aggregation=0.7, constraint\_chain=0.5, inverse\_reasoning=0.4.

**Empirical results** (5 batches × 5 QAs per cx level):

| cx | Generated | TOO_EASY | TOO_HARD | PASS |
|---|---|---|---|---|
| 0.10 | 25 | 3 | 11 | 10 (40%) |
| 0.40 | 25 | 0 | 6 | 19 (76%) |
| 0.70 | 25 | 2 | 9 | 13 (52%) |
| 0.95 | 25 | 0 | 6 | 19 (76%) |

## Configuration

### `configs/llm_config.yaml`

```yaml
providers:
  openai:
    api_key: ${OPENAI_API_KEY}
    default_model: gpt-4o
    # base_url: http://localhost:8000/v1   # uncomment for local vLLM
default_provider: openai
call_params:
  summary_agent:    {temperature: 0.3, max_tokens: 2000}
  item_writer:      {temperature: 0.7, max_tokens: 4000}
  solver_base:      {temperature: 0.6, max_tokens: 3000}
  solver_advanced:  {temperature: 0.6, max_tokens: 8000}
  judge:            {temperature: 0.1, max_tokens: 1000}
  graph_escalation: {temperature: 0.7, max_tokens: 4000}
```

To use a local model: set `base_url` to your vLLM endpoint. The `openai.AsyncOpenAI` client respects it with no other code changes. Known local-model issues: Qwen3 emits `<think>...</think>` tokens that break JSON extraction (strip before parsing); long escalation prompts (>30k tokens) need vLLM guided decoding (`response_format` with JSON schema).

### `configs/tool_config.yaml`

```yaml
enabled_tools: null   # null = all registered; or list names to restrict
search:    {provider: duckduckgo, max_results: 10}
scholar:   {api_key: ${SEMANTIC_SCHOLAR_API_KEY}, max_results: 5}
wikipedia: {max_length: 3000}
visit:     {timeout: 15, max_length: 5000}
table:     {timeout: 15, max_tables: 3, max_rows: 20}
```

## Extending the Pipeline

### Add a Tool

1. Create `src/tools/mytool.py`:
```python
from src.tools.tool_registry import BaseTool

class MyTool(BaseTool):
    name = "mytool"
    description = "One sentence for LLM. Usage: mytool | <query>"

    def __init__(self, config: dict = {}):
        self.config = config

    async def execute(self, query: str = "", **kwargs) -> dict:
        result = ...  # retrieval or computation
        return {"result": result, "cached": False}
```

2. Register in `scripts/generate_until_pass.py`:
```python
from src.tools.mytool import MyTool
registry.register(MyTool(cfg.get("mytool", {})))
```

3. Add config in `configs/tool_config.yaml`:
```yaml
mytool:
  api_key: ${MY_API_KEY}
```

The tool is immediately available to graph escalation and the advanced solver. The LLM discovers it via the `description` string injected into prompts — no prompt changes needed.

- **KG-construction only:** register it only in a custom `GraphBuilder` invocation, not in `generate_until_pass.py`.
- **QC solver only:** register normally but don't add to `GraphBuilder._pick_tools()`.
- **Caching:** cache at `data/.cache/{toolname}/{sha256(query)}.json`. See `wikipedia_tool.py` for the pattern.

### Add an Uncertainty Operation

In `src/graph/uncertainty_injector.py`, add one entry to `OPERATIONS`:
```python
OPERATIONS["my_operation"] = (
    "Precise instruction for the LLM: what to change, what to preserve."
)
```
No other changes. The operation is sampled uniformly alongside existing ones.

### Add an Escalation Strategy

1. Add the name string to `STRATEGIES` in `src/pipeline/stage2_graph_escalation.py`.
2. Add a description paragraph to `prompts/graph_escalation.txt` in the strategies section.
3. Add a complexity affinity (0.0=low cx preferred, 1.0=high cx preferred) to `_STRATEGY_AFFINITY` in `scripts/generate_until_pass.py`.

The LLM self-selects which strategy to apply based on the prompt descriptions; `parsed.strategy` records the choice in `qa.metadata["strategy"]`.

### Specialise for a Domain

1. Replace `data/seeds/seed_entities.json` with domain-specific seed topics.
2. Rebuild the KG: `python scripts/rebuild_kg.py`.
3. Optionally restrict tools: set `enabled_tools` in `tool_config.yaml` (e.g. `[scholar, wikipedia]` for academic domains).
4. Adjust uncertainty operations: remove operations irrelevant to your domain from `pipeline_config.yaml` `uncertainty_operations` list.
5. Adjust the judge prompt (`prompts/judge.txt`) for domain-specific answer formats (chemical formulas, numeric tolerances, etc.).

### Add an LLM Provider

In `src/utils/llm_client.py`:
```python
# In _init_clients:
if "myprovider" in providers:
    from myprovider_sdk import AsyncClient
    self._clients["myprovider"] = AsyncClient(api_key=providers["myprovider"]["api_key"])

# In _call_provider:
elif provider == "myprovider":
    return await self._call_myprovider(model, messages, params)
```

Follow `_call_openai` as the pattern: extract system messages, call API, return `str`. Set `default_provider: myprovider` in `llm_config.yaml`.

## Data Models

All inter-stage data is Pydantic v2 (`src/utils/data_models.py`), serialised via `model_dump()` / `model_validate_json()`.

**Key schemas:**

| Model | Key fields | Notes |
|---|---|---|
| `Entity` | `id, name, entity_type, attributes, tool_source, tool_query, snippet` | One atomic fact per node |
| `Relation` | `source_id, target_id, relation_type, evidence` | Typed directed edge |
| `QAPair` | `id, question, answer, difficulty, source_chunk_ids, escalation_history, metadata` | `source_chunk_ids` = KG entity IDs in subgraph |
| `Trajectory` | `qa_id, mode, steps, final_answer, tool_calls_used` | Full ReAct chain for every PASS |
| `TrajectoryStep` | `role, content, tool, tool_query, observation` | One turn (assistant or user/observation) |

`QAPair.metadata` carries: `strategy` (escalation strategy used), `complexity_level` (the `--complexity` value), `entities_incorporated` (names of newly added entities).

## Output Files

All output goes to `data/output/`. Files are **appended** across runs, not overwritten. Deduplication across runs is handled by the in-memory `SimilarityScorer` index (reset per run — cross-run deduplication requires loading prior embeddings manually).

| File | Format | Content |
|---|---|---|
| `passing_qas.jsonl` | JSONL | One `QAPair.model_dump()` per line |
| `passing_trajectories.jsonl` | JSONL | One `Trajectory.model_dump()` per line, for each PASS |
| `ui_data.json` | JSON | `{knowledge_graph, qa_pairs}` — viewer bundle, merged incrementally |
| `knowledge_graph.json` | JSON | `KnowledgeGraph.to_dict()` — reload with `KnowledgeGraph.from_dict()` |

## Known Issues

| Issue | Location | Fix |
|---|---|---|
| Noise nodes at high cx | `_ensure_connected` inserts KG artifact strings as connector nodes | Add name blocklist in `graph_builder.py` before `kg.add_entity` |
| Strategy monkeypatch not thread-safe | `generate_until_pass.py` replaces `STRATEGIES` list in-place | Pass `strategy_weights` as param to `escalate_batch` |
| Dedup O(n) in PASS count | `SimilarityScorer.is_duplicate` iterates all embeddings | Swap to `faiss.IndexFlatIP` above ~10k PASSes |
| DuckDuckGo rate limiting | `search_tool.py` retries 3× with 3s backoff | Set `BRAVE_API_KEY` to use Brave Search |
| Entity name resolution is exact-match | `_resolve_entity_ids` in escalation | Add `rapidfuzz.process.extractOne` fuzzy matching |
| Trajectories only on `generate_until_pass.py` path | `QualityController.evaluate_single` discards trajectory | Return trajectory from `evaluate_single` alongside verdict |

## Visualiser

`python ui/serve.py` starts an HTTP server on port 8765.

- `GET /ui/viewer.html` — D3.js KG viewer
- `POST /api/generate` — body `{complexity, batches, batch_size, api_key}` — spawns `generate_until_pass.py` subprocess
- `GET /api/status` — `{running, complexity, elapsed_s, log_tail[-20]}`

The viewer polls `/api/status` every 2.5s during generation and auto-reloads `ui_data.json` 1.2s after job completion. An `api_key` supplied via the UI takes priority over `env.sh`.
